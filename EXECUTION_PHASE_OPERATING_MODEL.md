## The New Session Framework

**Every session starts with one of these:**

### 1. 🔍 **Review Today's Code**

**What:** Review the actual repository against architecture

**Questions:**
- Does this code fit the 5-layer architecture?
- Are the layer boundaries respected?
- Is Diana a first-class citizen?
- Does this pass the 10-question PR checklist?
- What should we fix before merge?

**Outcome:** Code that strengthens the architecture

---

### 2. 🏗️ **Design One API**

**What:** Specify the contract for one capability

**Questions:**
- Which engine does this belong to?
- What are the inputs/outputs?
- How does Diana use this?
- What events does this emit?
- What permissions are required?

**Deliverable:** AIGINVEST Protocol spec for one capability

**Outcome:** Shared contract that developers can build against

---

### 3. 💻 **Implement One Service**

**What:** Write one complete service end-to-end

**Questions:**
- Which layer? Which engine?
- What's the happy path?
- What's the error path?
- How do we test it?
- How does Diana invoke it?

**Deliverable:** Service code + tests + deployment ready

**Outcome:** Working capability added to platform

---

### 4. 📋 **Review One Pull Request**

**What:** Review actual code using PR checklist

**Process:**
1. Developer opens PR
2. We walk through the 10 architectural questions
3. We review the code
4. We test the functionality
5. We either approve or request changes

**Outcome:** High-quality code that strengthens the platform

---

### 5. 🗓️ **Plan One Sprint**

**What:** Look at actual work for the next 2 weeks

**Questions:**
- What are the three priorities?
- Who owns each?
- What's the success criteria?
- What could block us?
- How do we measure progress?

**Deliverable:** Sprint board with clear tasks

**Outcome:** Team clarity on what matters next

---

### 6. 🚨 **Solve One Production Problem**

**What:** Real issues come up. We debug and fix them.

**Process:**
1. What broke?
2. Why did it break?
3. How do we fix it?
4. How do we prevent it next time?

**Outcome:** Reliable platform, team learning

## The Questions We Ask

### For Every Session:

1. **What are we building today?**
   - Specific, concrete outcome

2. **Why does this matter?**
   - How does it serve user progress?

3. **Where does it fit in the architecture?**
   - Which layer? Which engine?

4. **How will we know it works?**
   - What's the test?

5. **Who depends on this?**
   - What becomes possible next?

### For Every Code Review:

1. Which domain? (business area)
2. Which layer? (infrastructure/platform/mission/diana/experience)
3. 3 rules compliance? (diana/api/single responsibility)
4. Diana integration? (can she use it?)
5. Layer boundaries? (no cross-layer calls?)
6. Contract defined? (inputs/outputs clear?)
7. Tested? (unit/integration/e2e)
8. Observed? (metrics/logs/errors)
9. Secured? (auth/permissions/audit)
10. 10-year test? (still works at 10x scale?)
