## Part 4: The Engineering Compass

Every engineer should answer these five questions before starting any work:

### The Five Questions

**1. Which mission does this support?**
- Must map to a company mission
- If it doesn't, question whether it belongs in this sprint

**2. Which user problem does it solve?**
- Must be traceable to a user outcome
- Not a feature. An outcome.

**3. How will we know it's successful?**
- Must have measurable success criteria
- "Shipped" is not success. "Users use it and achieve outcomes" is success.

**4. How will Diana use it?**
- Will Diana's capabilities improve?
- Will this enable new Diana functionality?
- If neither, does it belong in v1.0?

**5. How will we test it?**
- Unit tests? Integration tests? User testing?
- What's the minimum test coverage?
- How do we know it won't break in production?

### The Compass in Practice

**Engineer's Code Review Self-Check (Before Opening PR):**

```
Before I submit this PR:

1. Mission: Which mission does this support?
   → Platform Foundation → Trust Engine → Event Logging
   
2. Problem: What user problem does this solve?
   → Users can see Diana's reasoning for recommendations
   
3. Success: How will we know it's successful?
   → 95%+ of AI actions have trust records
   → Users can click "Why?" on 100% of suggestions
   
4. Diana: How will Diana use this?
   → Diana can explain her reasoning to users
   → Diana can learn from user feedback on suggestions
   
5. Testing: How will we test this?
   → Unit test: TrustRecord creation
   → Integration test: Event → Trust record flow
   → Manual test: UI shows explanation
   → User test: Actual user can understand reasoning

If any answer is unclear, I refine my work before requesting review.
```

**Manager's Team Check-In (Weekly):**

```
Every Friday 10 AM:

For each engineer:

1. What mission are you working on?
2. What user outcomes are you targeting?
3. What blockers have appeared?
4. Do you have clarity on all 5 compass questions?
5. What do you need from me this week?

If anyone is blocked on compass clarity, we pause and align.
```

## Part 5: The Product Lifecycle

### Nine Stages (Nothing Skips)

Every feature goes through this exact sequence. No shortcuts.

```
Stage 1: IDEA
├─ User problem articulated
├─ Hypothesis formed
├─ Rough solution sketched
└─ Compass questions drafted

Stage 2: VALIDATED
├─ Problem validated with 3+ users
├─ Hypothesis confirmed or revised
├─ Solution approach agreed
└─ Compass questions answered

Stage 3: DESIGNED
├─ Wireframes created
├─ User flows documented
├─ Success metrics defined
├─ Diana integration points identified

Stage 4: IMPLEMENTED
├─ Code written
├─ Self-tested by engineer
├─ Ready for review
├─ All 5 compass questions answered

Stage 5: TESTED
├─ Unit tests passing (>90% coverage)
├─ Integration tests passing
├─ Code review approved
├─ Manual testing completed

Stage 6: RELEASED
├─ Deployed to staging
├─ Smoke tested
├─ Approved for production
├─ Deployed to production

Stage 7: MEASURED
├─ Usage metrics collected (1 week)
├─ User feedback gathered
├─ Success criteria evaluated
├─ Impact analyzed

Stage 8: IMPROVED
├─ User feedback incorporated
├─ Performance optimizations made
├─ Edge cases fixed
├─ Documentation updated

Stage 9: STANDARDIZED
├─ Pattern documented
├─ Other teams can replicate
├─ Added to company knowledge base
├─ Becomes future template
```

**Rules:**
- No feature can be called "done" until it reaches Stage 9
- Stages 1-2 are product/design (not engineering)
- Stages 4-6 are engineering (not product)
- Stages 7-9 are learning + standardization (everyone)
- Each stage has a clear completion criteria
- Skipping a stage requires explicit COO approval + documented reason

## Part 6: The Company Knowledge Base

### Four Permanent Collections

Every future employee starts by reading from these four categories.

#### 1. Decisions

*Why we made choices*

Example: "Why we chose PostgreSQL over MongoDB"
- Problem: Scale + consistency requirements
- Options: PostgreSQL, MongoDB, Cassandra
- Decision: PostgreSQL
- Reasoning: ACID compliance + JSON support + reasonable scaling
- Trade-offs: Lateral scaling more complex, but acceptable for v1.0
- Timestamp: July 7, 2026
- Maintainer: Engineering Lead
- Status: Locked (revisit if requirements change)

**Every major decision gets a Decision Record:**
- Problem statement
- Options considered
- Decision made
- Reasoning
- Trade-offs
- Date
- Maintainer
- Status (Locked, Under Review, Superseded)

#### 2. Architecture

*How the platform works*

Example: "Trust Engine Architecture"
- Purpose: Make every AI action transparent
- Components: Event model, TrustRecord model, UI layer
- Data flow: Action → Event → TrustRecord → UI
- Integration points: Diana service, User UI, Analytics
- Scalability: Handles 1000 events/min (tested to 10k/min)
- Monitoring: Event count, trust record latency, UI load time
- Future: Plan for event archival at 1M records

**Every major system component gets an Architecture Document:**
- Purpose
- Components
- Data flow
- Integration points
- Scalability notes
- Monitoring approach
- Future roadmap

#### 3. Product

*What users experience*

Example: "Mission Creation Flow"
- User objective: Create a new mission
- Steps: Click New Mission → Fill form → Auto-provision workspace → See workspace
- Success: User is in workspace ready to work
- Failure modes: Form validation error → show specific error
- Rollback: User can delete mission (soft delete)
- Metrics: Time from click to workspace (target: <30 seconds)

**Every user-facing feature gets a Product Document:**
- User objective
- Steps (UI flow)
- Success criteria
- Failure modes + handling
- Metrics + targets
- Rollback procedure

#### 4. Operations

*How the company runs*

Example: "Friday User Outcomes Ritual"
- Purpose: Ensure focus stays on outcomes, not features
- Participants: All team members
- Time: Every Friday, 4:00 PM - 4:30 PM
- Format:
  - Each team shares: "What did users accomplish?"
  - Not "What did we build?" but "What outcomes did our work enable?"
- Recording: Outcomes logged to public channel
- Cadence: Weekly, with monthly synthesis

**Every company process gets an Operations Document:**
- Purpose
- Participants
- Schedule
- Format
- Outputs
- How it connects to company missions

## Part 7: The AI Development Model

### How Engineering Works With Diana

```
Flow: Engineer → Diana → Engineer Review → Tests → Humans Approve → Deploy

Step 1: ENGINEER DEFINES
Engineer specifies: mission, problem, success criteria, compass questions

Step 2: DIANA PROPOSES
Diana generates code skeleton, tests, documentation, architecture

Step 3: ENGINEER REVIEWS
Engineer decides: approve as-is, request changes, or reject

Step 4: TESTS RUN
Automated tests run Diana's code (unit + integration + performance)

Step 5: HUMANS APPROVE
Team lead reviews for style, patterns, maintainability

Step 6: DEPLOY
Code goes to staging, then production, with automated monitoring

Step 7: DIANA LEARNS
Metrics collected, user feedback gathered, incorporated into future suggestions
```

**Rules:**
- Diana can propose. Only humans can approve.
- Every line of code is reviewed (AI-generated or not)
- Tests are never skipped (regardless of Diana's confidence)
- Production deploys require human approval (even if tests pass)
- Failures get recorded so Diana learns
- Success patterns get documented so others can replicate

**Example Diana Interaction:**

```

Inputs:
- Mission: Week 2 Trust Engine
- Problem: Log all platform actions for transparency
- Success: 95%+ AI actions have trust records
```

## Part 8: The 100-Year Principle

When making architectural decisions, ask:

> "Would we still be happy with this decision if AIGINVEST succeeds beyond our expectations?"

### What This Means

Not: "Can this scale to 1 million users?"

Instead: "Can this evolve with the company for decades?"

### Examples

**Database Choice (Passed 100-Year Test)**
- PostgreSQL is battle-tested for 20+ years
- Will scale to millions of rows
- Can handle 1000x growth before rebuilding
- Lots of expertise in ecosystem
- **Decision: Approved for v1.0 and beyond**

**Event Model (Passed 100-Year Test)**
- Immutable, append-only events enable audit trail forever
- Won't require rewriting history
- Enables future AI learning from company patterns
- Can eventually be archived (doesn't prevent scaling)
- **Decision: Approved as foundational**

**Microservices (Failed 100-Year Test)**
- Too fragmented for small team
- Hard to reason about long-term
- Would require significant ops overhead
- Better to monolith + modularize as team scales
- **Decision: Rejected for v1.0**

**Hard-Coded Configuration (Failed 100-Year Test)**
- Creates technical debt immediately
- Would need refactoring as company evolves
- Diana's behavior would be locked into code
- **Decision: Rejected. Use config system from start.**

### 100-Year Principle in Practice

**Team Discussion: Should we use no-SQL database?**

```
Engineer A: "No-SQL is faster for some queries"

Engineer B: "But it's less mature for structured data"

PM: "Let's use the 100-year principle. Would we still be happy 
     with a no-SQL choice if AIGINVEST becomes a major platform?"

Engineering Lead: "No. The transactional guarantees of SQL matter 
                  for financial + mission-critical data. A 20-year-old 
                  decision to use no-SQL would haunt us. Let's use PostgreSQL."

Decision: Approved. Team moves forward confident the choice works 
          for today AND tomorrow.
```

## Part 10: Three Parallel Tracks

These tracks run simultaneously and reinforce each other.

### Track 1: Product

*Deliver user-facing capabilities every sprint*

**Owned by:** Product + Design

**Cadence:** 1-week sprints

**Output:** Features that users ship and experience

**Success Metrics:**
- Feature shipped on schedule
- Users adopt feature
- Feature enables new user outcomes

**Examples:**
- Week 2: Trust Engine (users see why Diana recommends things)
- Week 3: Document Generator (users can create roadmaps)
- Week 4: Task Management (users track progress)

### Track 2: Platform

*Strengthen infrastructure, reliability, security, and scalability*

**Owned by:** Engineering

**Cadence:** 2-week cycles (offset from Product to provide stability)

**Output:** Reliable, secure, scalable systems

**Success Metrics:**
- Uptime: >99.95%
- Performance: P95 latency <100ms
- Security: 0 vulnerabilities
- Scalability: Can handle 10x current load

**Examples:**
- Database optimization (enable growth)
- API performance tuning (enable mobile)
- Security audit + remediation (enable enterprise)
- Disaster recovery testing (enable resilience)

### Track 3: Learning

*Collect user feedback, measure outcomes, and refine Diana based on evidence*

**Owned by:** Product + Engineering + Design

**Cadence:** Continuous (weekly synthesis, monthly deep dives)

**Output:** Evidence-based improvements

**Success Metrics:**
- User feedback collected and categorized
- Metrics tracked and analyzed
- Insights converted to improvements
- Diana learning incorporated

**Examples:**
- Weekly user interviews (what's working?)
- Usage analytics (how are users using features?)
- NPS surveys (would users recommend?)
- Diana behavior analysis (how is Diana helping?)

### How They Reinforce Each Other

```
Product Track: Delivers feature X
         ↓
Learning Track: Collects user feedback + measures impact
         ↓
Platform Track: Optimizes infrastructure based on usage patterns
         ↓
Product Track: Designs feature Y based on learning
         ↓
[cycle repeats, each iteration stronger]
```
